## 二、 实验目标

1. **动态场景渲染**：利用 Taichi 框架构建 3D 布料仿真场景，使用 GGUI 编写实时交互面板。
2. **质点-弹簧模型**：掌握基于物理的弹力与阻尼力计算方法，处理数值爆炸问题（速度钳制）。
3. **数值积分对比**：独立编写并比较显式欧拉、半隐式欧拉和隐式欧拉（定点迭代法）三种求解器，观察并分析其稳定性的内在差异。
4. **架构与性能优化**：通过数据场打平（一维化）与控制流重构，规避 Windows 环境下 LLVM 编译器的重定位 Bug。

## 三、 架构重构与物理原理

### 1. 物理计算优化：从 Scatter 到 Gather

为了避免多线程写入冲突（Race Conditions），本实验将传统的“散射（Scatter，遍历弹簧向两侧质点累加力）”架构重构为“聚集（Gather，遍历质点搜索邻居拉力）”架构。

* **优势**：免去了 `ti.atomic_add` 的原子操作开销；使受力计算与时间积分能够完整内联在有且仅有一个的并行循环中，大幅降低了 Kernel 启动开销。

### 2. 数值积分公式

* **显式欧拉 (Explicit Euler)**：

$$v_{t+1} = v_{t} + a_{t} \Delta t$$


$$x_{t+1} = x_{t} + v_{t} \Delta t$$


* **半隐式欧拉 (Semi-Implicit Euler)**：

$$v_{t+1} = v_{t} + a_{t} \Delta t$$


$$x_{t+1} = x_{t} + v_{t+1} \Delta t$$


* **隐式欧拉 (Implicit Euler)**：通过在 Python 侧构建对单步内核的松弛循环，实现定点迭代（Fixed-point Iteration）近似求解未来受力。

$$v_{t+1} = v_{t} + a_{t+1} \Delta t$$


$$x_{t+1} = x_{t} + v_{t+1} \Delta t$$

## 四、 实验核心：参数的演进与调控记录

本实验通过递进修改参数的方式，深入探究了物理参数（劲度系数、质量、阻尼）对仿真系统数值稳定性的影响。以下为参数演进的详细记录与物理分析：

### 阶段 1：初始参数状态 (Base Mode)

* **参数配置**：`k_s = 1500.0`, `mass = 1.0`, `k_d = 5.0`
* **现象观察**：
* **半隐式欧拉**和**隐式迭代**表现稳定，布料下落过程顺滑，并自然垂挂。
* **显式欧拉**表现出轻微的边缘抖动，但由于速度钳制（限制在 10.0 以内）的存在，系统并未发生完全发散。



### 阶段 2：劲度系数飙升与质量增加 (High-Energy Mode)

* **参数配置**：`k_s = 5000.0` (大幅提升), `mass = 1.5` (大幅提升), `k_d = 5.0` (保持原样)
* **更改意图**：大幅提升布料硬度，使其不易拉伸；同时增加质量以模拟更沉重的下坠惯性（重力 $F_g = m \cdot g$ 增大）。
* **现象观察与物理分析**：
* 由于弹簧劲度系数极高（$k_s = 5000.0$），质点稍有位移就会积蓄巨大的弹性势能，并在下一帧释放出极大的回复力。
* 此时，**显式欧拉法直接发生数值发散（爆炸）**。布料粒子在空中呈现无序炸开状，完全丧失了布料的几何形态，速度瞬间达到钳制上限。
* 实验结果证明，显式欧拉对硬约束（Stiff）系统的耐受度极低，其稳定边界（Stability Region）对时间步长 $\Delta t$ 的要求极其苛刻。



### 阶段 3：阻尼加倍与消能稳定 (Balanced Stiff Mode)

* **参数配置**：`k_s = 5000.0`, `mass = 1.5`, `k_d = 10.0` (大幅提升)
* **更改意图**：为了对抗阶段 2 中高硬度带来的能量激增，将阻尼系数加倍，强行在物理方程中引入更高的粘滞消能项（$f_d = -k_d \cdot v$）。
* **现象观察**：
* **半隐式欧拉**在此参数下表现出了极强的韧性，布料下落后的高频抖动被双倍阻尼迅速吸收，很快陷入静止，模拟出了高硬度厚重帆布的质感。
* **隐式迭代法 (4次迭代)** 表现最稳定，完全没有显式欧拉的无序振荡，证明了隐式方法在处理 stiff 问题上的无条件稳定特性（或者更宽的收敛半径）。
* **显式欧拉**在双倍阻尼的压制下，爆炸现象有所缓解，但依然存在明显的网格抽搐和持续抖动，无法陷入静止。

## 五、 环境 Bug 解决日志

在本地 Windows 平台（CPU 后端 `arch=x64`）运行过程中，遇到了编译期错误：
`LLVM Fatal Error: IMAGE_REL_AMD64_ADDR32NB relocation requires an ordered section layout`

* **原因剖析**：由于在单个 `@ti.kernel` 内部存在多个串行的并行循环，且包含了非静态的控制流（如隐式迭代的内部循环），超出了 Windows COFF 目标文件的 32 位相对寻址重定位限制。
* **解决方案**：
1. **数据打平**：将 `x` 和 `v` 的 Field 由二维形状 `(n, n)` 重构为一维 `(n * n)`，降低了控制流维度，并完美适配了 GGUI 的 `scene.particles()` 渲染引擎。
2. **控制流分离**：将隐式迭代的外部循环移至 Python 侧，Kernel 只负责纯粹的单步松弛迭代，从而彻底消除了 LLVM 编译器的重定位 Bug。
